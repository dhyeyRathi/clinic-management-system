# **🏢 1. Project Overview**
The **Clinic Management System** is a secure, role-based, multi-user healthcare workflow platform. It digitizes operations by centralizing client scheduling, doctor consultations, receptionist queue management, medical histories, prescription handling, laboratory test bookings, lab result distribution, and manager dashboards. The platform ensures strict data privacy and isolation using a robust Role-Based Access Control (RBAC) model.
---
# **🛑 2. Problem Statement**
Traditional clinic operations rely heavily on manual procedures, which suffer from major operational bottlenecks:
- **Manual Scheduling Errors:** Double-bookings, slot conflicts, and inefficient doctor availability tracking.
- **Disconnected Medical Records:** Scattered paper histories and lost diagnostic reports make tracking client history difficult.
- **Prescription Loss:** Verbal or handwritten prescriptions are easily lost, raising compliance and safety risks.
- **Manual Lab Booking Inefficiencies:** Delayed test requests, lost physical result sheets, and poor tracking of scheduled diagnostics.
- **Data Security & Privacy:** Administrative staff having access to sensitive clinical notes, or clients viewing other clients' records.
- **Lack of Audits:** Inability to track system overrides, financial logs, or role-based account activities.
---
# **👥 3. Target Users**
- **👤 Clients:** Individuals seeking to book clinic sessions, request lab tests, download prescriptions, view history, and receive notifications.
- **🩺 Doctors:** Healthcare practitioners managing assigned schedules, consults, clinical notes, recommending lab tests, and issuing digital prescriptions.
- **🛎️ Receptionists:** Front-desk operators managing schedules, queue check-ins, client registration, and appointment confirmation.
- **🥼 Lab Managers:** Laboratory supervisors responsible for confirming lab bookings, managing lab tests availability, and uploading results.
- **💼 Managers:** Administrators overlooking staff configurations, clinic analytics, resource usage (rooms/lab equipment), and audit logs.
---
# **📊 4. Research**
During the portfolio planning stage, three distinct project concepts were evaluated for technical viability, business value, and portfolio impact:
- *Enterprise Asset and Resource Management System*
- ***Clinic Appointment and Client Management System***
- *Internship Supervision Task Management System*
The comparison of features indicated that the Clinic Management System presented the highest implementation difficulty, but also the greatest real-world utility and portfolio impact. It demands a sophisticated RBAC architecture, strict field-level data privacy, and a real-time workflow coordination engine, making it the most suitable candidate for full-stack software development.

# **👥 7. User Roles**
## **👤 7.1 Client**
**Purpose:** Request appointments, book lab tests, track personal medical timeline, and download prescriptions.
- **Allowed Actions:** Register/Login, update self-profile, book appointments, view own records (prescriptions, history, invoices, lab reports), book lab tests (status set to pending), cancel bookings.
- **Restricted Actions:** Cannot view other clients, create/edit prescriptions, edit clinical logs, manage lab test service catalogs, or view manager finance dashboards.
## **🩺 7.2 Doctor**
**Purpose:** Manage schedules, record consultation notes, recommend lab tests, and author prescriptions.
- **Allowed Actions:** View assigned appointments, read assigned client histories, record consultation outcomes and notes, recommend lab tests, issue prescriptions, upload scanned reports.
- **Restricted Actions:** Cannot see global clinic revenue reports, modify staff accounts, approve or reschedule lab bookings, or view clients outside of assigned consultations.
## **🛎️ 7.3 Receptionist**
**Purpose:** Coordinate front-desk schedule flow and queue operations.
- **Allowed Actions:** View doctor calendars, register new clients, confirm/reject/reschedule appointments, manage check-in queues, assign clients to doctors.
- **Restricted Actions:** Cannot read clinical notes, view or download client prescriptions, manage lab bookings, access diagnostic medical files, or change system roles.
## **🥼 7.4 Lab Manager**
**Purpose:** Supervise laboratory bookings, check-in clients for diagnostic tests, and upload results.
- **Allowed Actions:** View laboratory schedule calendars, confirm/reject/reschedule lab bookings, manage test-specific availability status in the catalog, upload result PDFs/images to Cloudinary and link them to lab bookings.
- **Restricted Actions:** Cannot read clinical notes, view or download client prescriptions, schedule doctor appointments, view global clinic revenue, or modify user accounts.
## **💼 7.5 Manager**
**Purpose:** Oversee staff, resources, lab catalogs, finances, and clinic analytics.
- **Allowed Actions:** Manage staff accounts (CRUD), manage rooms, lab equipment, and service catalogs (CRUD for Lab Tests), override schedule conflicts, view finance reports, and audit system activity logs.
- **Restricted Actions:** Cannot edit client clinical notes or alter medical history records.
## **🛡️ 7.6 Role Creation Pathways & Security Constraints**
To prevent privilege escalation and enforce strict role boundaries, user creation is governed by database-level triggers and RLS guards:
- **Anonymous Public Signups:** Restricted exclusively to the `CLIENT` role. Any attempt to register other roles during public signup will abort the transaction.
- **Staff Registrations:** Only an authenticated user with the `MANAGER` role can create user accounts. The manager is strictly limited to creating `DOCTOR`, `RECEPTIONIST`, and `LAB_MANAGER` accounts. The manager cannot create `CLIENT` or `MANAGER` accounts.
- **Manager Accounts:** Created exclusively through direct database querying or seeding. They cannot be created via the frontend application.

# **🔄 8. User Flow**
## **8.1 Appointment Booking Flow**
1. Client logs in, searches for doctors by specialization, and requests a time slot.
2. Receptionist receives the booking in the incoming queue, checks doctor availability, and confirms the request.
3. System triggers an in-app booking confirmation notification to the client.
4. The slot is locked, updating the doctor's daily schedule calendar.
## **8.2 Consultation & Prescription Flow**
1. Client arrives at the clinic; Receptionist marks the client as Checked In.
2. The queue dashboard updates, moving the client to the doctor's active waiting queue.
3. Doctor opens the appointment from their dashboard, reviews the client's medical timeline, and enters clinical notes.
4. Doctor enters prescription details (dosage/medications) and optionally recommends a list of lab tests.
5. Doctor marks the session Completed and requests a follow-up date.
6. Client logs in to download the prescription PDF.
## **8.3 Lab Test Booking & Results Flow**
1. Client views recommended lab tests from their consultation timeline.
2. Client schedules a lab booking, selecting a test type and slot.
3. Lab Manager receives the booking, checks resource slots, and confirms the lab appointment.
4. Client arrives for the test; Lab Manager marks the client as Checked In.
5. Lab Manager uploads the completed result PDF to Cloudinary and links it to the lab booking.
6. System marks the booking Completed and triggers an alert notifying both the Client and recommending Doctor that results are ready.
## **8.4 Manager Auditing Flow**
1. Manager logs in and views the consolidated operations panel.
2. Manager checks financial collections (invoiced vs. paid amounts) and doctor/lab utilization.
3. Manager opens the system audit panel to review sensitive events, such as appointment overrides or staff role changes.

# **💾 10. Database Design**
## **Table Schema: User Profiles**
- `id` (UUID, PK, references auth.users)
- `name` (String)
- `phone` (String)
- `role` (Enum: CLIENT, DOCTOR, RECEPTIONIST, LAB_MANAGER, MANAGER)
- `status` (Enum: ACTIVE, INACTIVE, SUSPENDED)
- `avatar_url` (String)
- `created_at` (Timestamp)
- `updated_at` (Timestamp)

## **Table Schema: ClientProfile**
- `id` (UUID, PK)
- `user_id` (UUID, FK profiles.id)
- `client_code` (String, Unique MRN format: PAT000001)
- `date_of_birth` (Date)
- `gender` (String)
- `address` (Text)
- `emergency_contact` (String)
- `medical_notes_summary` (Text)

## **Table Schema: DoctorProfile**
- `id` (UUID, PK)
- `user_id` (UUID, FK profiles.id)
- `doctor_code` (String, Unique MRN format: DOC000001)
- `specialization` (String)
- `qualifications` (String)
- `availability` (JSON weekly slots schedule)
- `consultation_fee` (Decimal)

## **Table Schema: Appointment**

- `client_id` (UUID, FK client_profiles.id)

## **Table Schema: LabTestTypes**

# **🔗 11. Entity Relationships**
- **User to Profiles:** One-to-One relationship between User and ClientProfile or DoctorProfile.
- **Client/Doctor to Appointment:** One-to-Many relationship where profiles contain multiple Appointments.
- **Invoice & Invoice Items:** One-to-Many relationship between Invoice and InvoiceItem. The InvoiceItem references transactional records (appointments or lab_test_types) using polymorphic referenceId bindings.
- **Appointment to Clinical Records:** One-to-One relationship between Appointment and VisitHistory, Prescription, or Invoice.
- **Client to Documents:** One-to-Many relationship between ClientProfile and Reports or Prescription (stored in Cloudinary).
- **Actor to ActivityLogs:** One-to-Many relationship where a User acts as the author of multiple ActivityLogs.

# **🔑 14. Authentication Approach**
- **Supabase Auth:** Email/password logins managed securely with cookies.
- **Session Middleware:** Next.js Server-Side Middleware reads cookies to enforce page protection layouts.
- **User Metadata Claims:** Session JWT encodes user roles to dynamically route dashboards.
- **Database RLS Integration:** Supabase Row-Level Security checks verify authentications directly inside PostgreSQL query processes.
---
# **🛡️ 15. Validation Strategy**
- **Client Schema Validation:** Forms utilize Zod schemas with React Hook Form to reject illegal inputs before submission.
- **API Schema Verification:** Serverless Next.js API endpoints run payload validations via Zod schemas.
- **Database Constraints:** Hard constraint rules (e.g. check constraints, unique fields, primary keys) prevent corrupt records.

# **⚠️ 16. Error Handling Strategy**
- **API Responses:** Standardized JSON payload errors containing visual debug values.
- **Toast Notifications:** Client-side API catch-blocks emit toast notifications immediately.
- **React Boundaries:** Complex scheduling and board screens run inside localized Error Boundary components.
---
# **🔒 17. Security Considerations**
- **PostgreSQL Row-Level Security (RLS):** Policies secure table-level read/writes. Example:
```sql
CREATE POLICY client_own_history ON VisitHistory FOR SELECT USING (auth.uid() = (SELECT userId FROM ClientProfile WHERE ClientProfile.id = clientId));
```
- **Cloudinary Short-Lived Upload Signatures:** Backend API generates cryptographic hashes so that direct client-side uploads are signed and restricted.
- **XSS Protection:** Next.js framework renders code safely, preventing dangerous HTML injection.

# **⚠️ 23. Risks and Solutions**
- **Risk: Unauthorized Access to Client Records**
  Solution: Activate Supabase Row-Level Security (RLS) on all client tables and enforce authentication policies using auth.uid() matching. Lab reports are isolated from administrative files.
- **Risk: Appointment & Lab Slot Overlapping/Double Booking**
  Solution: Execute database-level lock checks on PostgreSQL triggers to prevent duplicate slots during transactional confirmations.
- **Risk: Cloudinary Asset Upload Security Vulnerabilities**
  Solution: Direct client uploads to Cloudinary are forbidden. Client must retrieve a cryptographically signed signature from our secure backend API route first.
